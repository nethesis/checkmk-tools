DESCRIPTION:
Advanced ransomware detection script for Windows Server. Monitor shares 
network (UNC or local) looking for suspicious activity indicative of an attack 
Current or recent ransomware.

MAIN FEATURES:
 Detection of files with ransomware extensions (50+ known variants)
 File entropy analysis (encrypted files have entropy >7.5)
 Ransom notes detection (README, DECRYPT, HOW_TO, etc.)
 Monitoring of anomalous extension changes
 Canary files (decoy files for early detection)
 Massive encryption activity analysis
 Suspicious login patterns
 Timeout and performance optimization for large shares
 External JSON configuration
 Statefile for continuous tracking

DETECTION LOGIC:

1. SUSPICIUS FILES SCORING:
   - Files with ransomware extension: +10 points
   - High entropy (>7.5): +5 points
   - Recent extension change: +3 points
   - Abnormal file name (>100 chars or strange characters): +2 points
   - Score ≥5 → Suspicious file

2. MASS ACTIVITY DETECTION:
   - Modified files > AlertThreshold in TimeWindow → CRITICAL
   - Change Rate > MaxFilesPerSecond → CRITICAL
   - Multiple shares hit simultaneously → CRITICAL

3. CANARY FILES:
   - Hidden files automatically created in shares
   - Attributes: Hidden, ReadOnly
- If modified/deleted → CRITICAL ALERT
   - Early detection before extensive damage

4. RANSOM NOTES:
   - Pattern matching on common file names
   - Keyword analysis (ransom, decrypt, bitcoin, payment, crypto)
   - Presence of ≥2 keywords in file → Ransom note detected

ENTROPY ALGORITHM:
Calculate Shannon entropy on 1024 byte sample of the file:
- Text file: 3-5 bits/byte
- Compressed files: 6-7 bits/byte
- Encrypted files: >7.5 bits/byte
Formula: H = -Σ(p(i) * log₂(p(i)))

FEATURES:
- Name: .canary_donotdelete.txt (configurable)
- Attributes: Hidden, ReadOnly
- Content: Warning message with metadata
- Location: Root of each monitored share
- Automatic creation on first run

TYPICAL CONTENT:
```
WARNING: This is a ransomware tracking file.
DO NOT DELETE, MODIFY OR MOVE this file.

This file is used to detect suspicious ransomware activity.
Changing or deleting it will generate a security alarm.

Creation date: 2025-10-27 14:30:00
Share: \\SERVER\Documents
System: WS2022AD
```

MONITORING:
1. Script checks for canary existence at each run
2. If missing → CRITICAL alert
3. If modified (LastWriteTime < 30 min) → CRITICAL alert
4. Store hash in statefile for integrity check

ADVANTAGES:
 Early detection (before damage is extensive)
 Zero-day ransomware detection
 It doesn't depend on signature
 Low overhead performance
 Difficult to evade for ransomware

LIMITATIONS:
 Advanced ransomware can recognize canaries
 Requires share write permissions
 It can generate false positives if users touch files


========================================================================================
ADVANCED ALGORITHMS
========================================================================================

1. ENTROPY ANALYSIS (Shannon Entropy)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Formula: H = -Σ(p(i) × log₂(p(i)))

Where:
- p(i) = frequency of byte i in the file
- H = entropy in bits/byte (0-8)

Interpretation:
- 0-3 bits: Very repetitive data (text, log)
- 3-5 bits: Normal data (documents, uncompressed images)
- 5-7 bits: Compressed data (ZIP, JPEG)
- 7-8 bits: Random or encrypted data (ENCRYPTED)

Threshold: >7.5 bits → Suspicious file encrypted

Performance:
- Sample first 1024 bytes only
- O(n) calculation on sample
- ~1ms per file


2. SUSPICIUS FILES SCORING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Cumulative Score:
+10 points: Extension in RANSOMWARE_EXTENSIONS
+5 points: Entropy >7.5
+3 points: Recent extension change (different backup extension)
+2 points: Abnormal file name (>100 chars or non-ASCII)

Threshold: Score ≥5 → Suspicious file

Example:
File: "document.docx.locked" (LastWrite: 5 min ago)
- .locked extension: +10
- Entropy 7.8: +5
- Normal name: +0
Total score: 15 → HIGHLY SUSPICIUS


3. MASS ACTIVITY DETECTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Mass alert conditions:
1. Edited Files > AlertThreshold in TimeWindow
2. Change Rate > MaxFilesPerSecond
3. Multiple shares affected simultaneously

Rate calculation:
Rate = SuspiciousFilesCount / (TimeWindowMinutes × 60)

Example:
- 150 suspicious files in 30 minutes
- Rate = 150 / 1800 = 0.083 files/sec
- If MaxFilesPerSecond = 10 → OK
- But 150 > AlertThreshold (50) → CRITICAL

Logic:
If (Count > Threshold) OR (Rate > MaxRate) → Mass Activity


4. PATTERN MATCHING RANSOM NOTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Step 1: File name matching
Patterns: README*, DECRYPT*, HOW_TO*, *.hta, *.html

Step 2: Content analysis
Keywords: [ransom, decrypt, bitcoin, payment, crypto, locked, encrypted]

Step 3: Scoring
- File created <24h: eligible
- Keyword count ≥2: Identified as ransom note
- Unreadable but recent file: Suspicious

Example:

TROUBLESHOOTING
========================================================================================

PROBLEM: Script does not generate output
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CAUSE: Execution policy or syntax error
SOLUTION:
1. Check execution policy:
   Get-ExecutionPolicy -List
2. Set RemoteSigned:
   Set-ExecutionPolicy RemoteSigned -Scope CurrentUser
3. Test manually:
powershell -ExecutionPolicy Bypass -File check_ransomware_activity.ps1
4. Check for errors:
   powershell -File check_ransomware_activity.ps1 -VerboseLog


PROBLEM: "Access denied" on UNC share
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CAUSE: CheckMK agent runs as SYSTEM, has no network access
SOLUTION:
1. Check manual access:
   Test-Path "\\SERVER\Share"
2. CheckMK agent must run as a service account with share access
3. Configure service accounts:
   services.msc → CheckMK Service → Log On → Account with share access
4. Alternative: use local path if share on same server


PROBLEM: Canary file is not created
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CAUSE: Insufficient write permissions
SOLUTION:
1. Check CheckMK account permissions:
   Test-Path "\\SERVER\Share" -PathType Container
2. Test writing:
   "test" | Set-Content "\\SERVER\Share\test.txt"
3. Check share permissions (SMB):
   - Read/Write for CheckMK accounts
4. Check NTFS permissions:
   - Modify for CheckMK account
5. Disable canary if not needed:
   "EnableCanaryFiles": false in config


PROBLEM: Slow performance / timeouts
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CAUSE: Share very large or slow
SOLUTION:
1. Reduce MaxFilesToScan in config:
   "MaxFilesToScan": 5000
2. Increase ExcludePaths:
   "ExcludePaths": ["backup", "archive", "old", "temp"]
3. Reduce TimeWindowMinutes:
   "TimeWindowMinutes": 15
4. Script has built-in 30 sec timeout, but can be increased in code


PROBLEM: Too many false positives
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CAUSE: Threshold too sensitive
SOLUTION:
1. Increase AlertThreshold:
   "AlertThreshold": 100
2. Increase MaxFilesPerSecond:
   "MaxFilesPerSecond": 20
3. Exclude folders with high activity:
   "ExcludePaths": ["working", "processing", "import"]
4. Remove custom extensions that generate false positives


PROBLEM: Config file not found
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CAUSE: Non-standard path or typo
SOLUTION:
1. Script searches in 3 locations automatically
2. Specify explicit path:
   check_ransomware_activity.ps1 -ConfigFile "C:\custom\config.json"
3. Check exact file name:
   dir "C:\ProgramData\checkmk\agent\local\ransomware_config.json"
4. Create if missing:
   '{"SharePaths":["\\\\SERVER\\Share"]}' | Set-Content config.json


PROBLEM: Remote Wrapper does not download scripts
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CAUSE: Firewall, proxy, or GitHub down
SOLUTION:
1. Test GitHub connectivity:
   Test-NetConnection raw.githubusercontent.com -Port 443
2. Test manual download:
   Invoke-WebRequest https://raw.githubusercontent.com/... -OutFile test.ps1
3. Check proxy settings if present
4. Check firewall outbound rules
5. Fallback: Use local script instead of wrappers


PROBLEM: Output with corrupted characters
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CAUSE: Console encoding vs CheckMK
SOLUTION:
1. Script uses "task" instead of "activity" (fix applied)
2. Avoid accented characters in output
3. Use [Console]::OutputEncoding = [System.Text.Encoding]::UTF8

SAFETY
========================================================================================

CONSIDERATIONS:

RANSOMWARE DETECTION:
 Script reads files but does not write (except canary and statefile)
 No changes to monitored files
 Read-only access sufficient (except canary)
 Does not execute files or commands from shares

REMOTE WRAPPER:
 Download and run code from GitHub
 MITM risk if network compromised
 Check HTTPS and certificate
 Production use: Local script preferred

CREDENTIALS:
 Script does not store credentials
 Use context account CheckMK service
 No hardcoded passwords
 Config file does not contain secrets

MINIMUM PERMITS:
- Read on monitored shares
- Write only for canary files (optional)
- Write in %TEMP% for statefile

LOG AND STATEFILE:
 Statefile contains only metadata (no file content)
 Do not log file contents
 IP and path share visible in CheckMK output
 It does not expose sensitive business information

CANARY FILES:
 Harmless content (warning message only)
 Hidden attributes to reduce visibility
ReadOnly to prevent accidental editing
 It does not contain sensitive data


SECURITY BEST PRACTICES:
1. Run CheckMK service with dedicated account (not SYSTEM)
2. Limit account permissions to only necessary shares
3. Configure firewall to block outbound except GitHub (if wrapper)
4. Monitor statefiles for tampering
5. Share access log audit
6. Periodic rotation of canary files (hash verification)
7. Immediate alerts on CRITICAL (SIEM integration)
8. Frequent and tested backups for disaster recovery
